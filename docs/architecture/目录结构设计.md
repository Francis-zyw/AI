# 项目目录结构设计

## 1. 设计目标

为了方便后续继续开发 OCR、章节解析、表格抽取、构件识别和规则引擎，目录结构按“代码、数据、文档、脚本、测试、历史原型”分层整理。

## 2. 当前推荐结构

```text
智能提量工具/
├── data/
│   ├── input/                # 输入 PDF、图片、样例文件、构件库主数据
│   ├── output/
│   │   └── step1/            # 第一步章节识别输出
│   ├── cache/                # 运行缓存
│   ├── temp/                 # 临时文件
│   └── logs/                 # 日志
├── docs/
│   ├── architecture/         # 架构设计和目录设计
│   ├── plans/                # 流程方案、阶段计划
│   └── step1/                # 第一步相关说明文档
├── scripts/                  # 双击运行脚本、启动脚本
├── step1_chapter_ocr/        # 当前主线代码包
├── tools/
│   ├── tool_component_match_review/ # 构件匹配结果人工修订工具
│   └── tool_component_type_library/ # 构件类型-属性库维护工具
├── tests/
│   ├── unit/                 # 单元测试
│   ├── integration/          # 集成测试
│   ├── fixtures/             # 测试样例
│   └── manual/               # 人工验证记录或手工样例
├── legacy/
│   ├── prototypes/           # 历史原型代码
│   └── reference_data/       # 旧示例和历史参考数据
└── 分析工具/venv/            # 当前运行环境依赖
```

## 3. 各层职责

### 3.1 `step1_chapter_ocr/`

当前第一步主线代码放在这里，保持可以直接 `import` 调用。

建议后续继续在这个包里扩展：

- `api.py`：统一入口
- `core.py`：主流程
- `providers.py`：OCR / 文本抽取适配层
- `models.py`：结构化结果模型

### 3.2 `data/`

所有输入、输出、缓存和日志统一放在 `data/` 下，避免测试过程中到处散落结果文件。

其中 `data/input/` 现在同时承担两类角色：

- 主流程原始输入，例如 PDF、图片、样例文件
- 可被主流程直接消费的基础库数据，例如 `components.json`、`components.jsonl`

### 3.3 `docs/`

流程说明、结构说明、交付文档统一放在 `docs/` 下，代码目录保持更干净。

### 3.4 `scripts/`

双击运行脚本集中到这里，后续如果新增：

- 批量处理脚本
- OCR 安装脚本
- 数据清洗脚本

当前命名建议：

- 主流程脚本使用 `stepN_功能名`，例如 `step1_select_pdf_mac.command`
- 辅助工具脚本使用 `tool_工具名`，例如 `tool_component_match_review.bat`

也都建议放在这个目录。

### 3.5 `tools/`

周期性维护但不属于主解析链路核心步骤的工具，建议集中到 `tools/` 下。

当前已纳入：

- `tools/tool_component_type_library/`
  - 用于维护构件类型-属性库
  - 默认读取 `data/input/component_type_attribute_excels/`
  - 默认输出 `data/input/components.json` 和 `data/input/components.jsonl`

- `tools/tool_component_match_review/`
  - 用于查看和人工修订 Step 2 的匹配结果

### 3.6 `tests/`

后续如果继续开发：

- 章节边界识别
- OCR fallback
- 表格切分
- 构件匹配

都建议配套放进 `tests/`，便于长期维护。

### 3.7 `legacy/`

历史原型和旧参考数据先隔离保存，不放在主线开发区，避免影响后续目录清晰度。

## 4. 后续扩展建议

建议未来功能按阶段继续扩展：

1. `step1_chapter_ocr/`：正文范围识别、章节切分、OCR fallback
2. `step2_table_extract/`：章节内表格区域和表格结构抽取
3. `step3_rule_parse/`：清单项、特征、计量单位、规则字段抽取
4. `step4_component_match/`：构件和关键词归类

如果后续阶段变多，也可以再升级为：

```text
src/
  step1_chapter_ocr/
  step2_table_extract/
  step3_rule_parse/
```

但在当前阶段，继续保留 `step1_chapter_ocr/` 在项目根目录，代码调用会更简单。
