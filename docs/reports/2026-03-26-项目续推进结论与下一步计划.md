## 三、当前最主要未完成事项

### 1. Step3 尚未进入正式模型闭环

现状：

- `step3_filter_condition_match/runtime_config.ini` 默认 `local_only = true`
- shell 环境中未检测到 `OPENAI_API_KEY` / `OPENAI_BASE_URL`

这意味着：

- 即使代码正确，按当前默认配置运行，也只会得到本地规则结果
- 不会进入正式的模型校正阶段
- 因而无法形成“可交付正式结果”

### 2. Step2 虽有输出，但仍处于 partial 状态

需要进一步核查：

- 是否已完成所有章节与批次
- 是否存在需要重跑或补合成的章节
- `component_matching_result.json` / `synonym_library.json` 是否满足 Step3 正式输入要求

### 3. Step3 Review Queue 仍未显式落地

虽然 V2 设计中已经把 `candidate_only / unmatched / conflict` 识别为 review queue 候选，但当前项目内仍需补：

- review queue 标准目录
- review ledger 结构
- 回写链路

### 4. Cutover 标准仍未真正执行

目前已有 cutover gate 文档，但尚未完成：

- baseline 对比
- shadow run
- 人工抽检
- go/no-go 判定

## 四、建议的下一阶段执行顺序

### P0：打通 Step3 正式模型运行环境

目标：让 Step3 从 `completed_local_only` 升级为正式模型闭环。

动作：

1. 不在仓库写入真实密钥。
2. 通过环境变量或统一运行入口注入：
   - `OPENAI_API_KEY`
   - `OPENAI_BASE_URL`（如有代理/兼容层）
3. 明确正式运行时将 `local_only=false`。
4. 先对小样本跑一次 Step3 正式模式验证。
5. 再决定是否全量运行。

### P1：校验并必要时修复 Step2 主结果完整性

目标：确认 Step3 正式运行所依赖的 Step2 结果是可信且完整的。

动作：

1. 校验 `run_summary.json.status`
2. 核查已完成章节/批次数
3. 如需，使用 `step2-synthesize` 或 `step2-execute` 做补合成/补跑
4. 固化一版“Step2 可作为 Step3 正式输入”的检查报告

### P2：显式落地 Step3 Review Queue 与人工回写契约

目标：把当前 backlog 变成可追踪工作面，而不是悬空统计数字。

动作：

1. 建立 `data/manual_reviews/step3/`
2. 输出标准字段：
   - `record_id`
   - `row_id`
   - `quantity_component`
   - `calculation_item_code`
   - `review_status`
   - `manual_notes`
   - `reviewed_at`
3. 设计从 review ledger 回写到 final export 的链路

### P3：在 Step3 正式结果上复核 Step4 是否需要重跑

目标：避免 Step4 长时间依赖 Step3 的半成品输入。

动作：

1. 对比当前 `data/output/step4/` 与新的 Step3 正式结果
2. 判断哪些构件类型需要重跑
3. 必要时只对受影响 component groups 做增量重跑

### P4：准备 cutover 验收

动作：

1. baseline 对比
2. shadow run
3. 人工抽检
4. cutover 报告

## 五、当前建议结论

如果只选一个最优先事项，当前应优先推进：

> **先把 Step3 的正式模型闭环打通。**

原因：

- Step2/3/4 代码和测试已经基本可用
- 现在最关键的断点不是“有没有框架”，而是“正式模式没跑起来”
- Step3 不闭环，Step4 就难以成为真正可信的最终结果

## 六、后续接手提示

后续继续推进时，不建议再从“重做方案”开始讨论，而应直接按下面顺序执行：

1. 凭据与运行模式注入
2. Step3 小样本正式跑通
3. Step2 完整性核查
4. Step3 全量/半量正式运行
5. Review queue 落地
6. Step4 增量重跑与验收

这会比继续停留在概念性方案阶段更高效。
