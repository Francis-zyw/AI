## 4. 功能总览

本工具用于对国标类 PDF 文档进行逐步结构化解析，目标是把原始标准文档逐层转换成可用于构件识别、规则提取、清单匹配和自动提量的结构化中间结果。

当前主线能力包括：

1. Step1：从 PDF 中识别目录、正文范围、章节与表格区域。
2. Step2：把构件库中的构件名称匹配到国标中的标准构件名称，并沉淀同义词库。
3. Step3：结合 Step1 表格区域与 Step2 同义词库，识别清单项、构件类型、项目特征表达式和计算项目。
4. Step4：在指定构件类型的前提下，对清单项做本地直匹配和模型精修。
5. 审计与计划：支持对现状进行 `audit` 与 `plan` 输出。
6. 兼容导入：支持把历史 Step1、Step2、Step3 结果导入到当前 V2 工作区。
7. 人工复核：支持从 Step3 结果生成 review queue。

## 8. Step1 使用说明

### 8.1 功能说明

Step1 用于从 PDF 中识别目录和正文有效范围，按章节递归切分文本，并额外提取含表格的区域，作为 Step2 和 Step3 的输入基础。

### 8.2 命令

```bash
python3 -m pipeline_v2 step1-extract \
  --pdf data/input/房屋建筑与装饰工程工程量计算标准.pdf \
  --output data/output/step1/run-20260407-step1
```

### 8.3 常用参数

- `--pdf`
  - 必填，PDF 路径。
- `--output`
  - 可选，输出目录；默认会写到 `data/output/step1/<pdf-name>`。
- `--no-save`
  - 只返回摘要，不写文件。
- `--use-paddleocr`
  - 打开 PaddleOCR fallback。
- `--no-auto-install-paddleocr`
  - 禁止自动安装 PaddleOCR。
- `--minimum-text-length`
  - 文本层回退阈值。
- `--write`
  - 把 CLI 返回摘要额外写入一个 JSON 文件。

### 8.4 输入

- 一个 PDF 文件。

### 8.5 输出

Step1 典型输出文件：

- `catalog_summary.json`
- `outline_entries.json`
- `flat_regions.json`
- `region_tree.json`
- `table_regions.json`
- `chapter_regions/chapter_index.json`
- `chapter_regions/<chapter_slug>.json`
- `region_texts/*.txt`

### 8.6 输出用途

- `chapter_regions/chapter_index.json`
  - 供 Step2 读取章节索引。
- `table_regions.json`
  - 供 Step3 读取表格区域。
- `flat_regions.json`
  - 供结构审计、回溯和二次处理使用。

### 8.7 Step1 返回摘要字段

CLI 直接输出的是 `summary`，通常包含：

- `pdf_path`
- `total_pdf_pages`
- `catalog_detection`
- `outline_counts`
- `region_counts`
- `table_counts`
- `providers`
- `output_dir`

### 8.8 Step1 Python 调用示例

```python
from pipeline_v2.step1_chapter_ocr.api import process_pdf

result = process_pdf(
    pdf_path='data/input/房屋建筑与装饰工程工程量计算标准.pdf',
    output_dir='data/output/step1/run-demo',
    save_outputs=True,
    use_paddleocr=False,

print(result.to_dict()['summary'])
```

## 9. Step2 使用说明

### 9.1 功能说明

Step2 用于把构件库中的构件名称与国标章节、表格和术语进行匹配，输出：

- 构件名称与标准名称的映射关系。
- 同义词库 `synonym_library.json`。
- 后续 Step3 可直接消费的结构化结果。

### 9.2 Step2 的三个命令

#### 9.2.1 `step2-prepare`

只做准备，不发模型请求。

```bash
python3 -m pipeline_v2 step2-prepare \
  --components data/input/components.json \
  --step1-source data/output/step1/run-20260407-step1 \
  --output data/output/step2/run-20260407-step2-prepare
```

输出：

- `prepare_manifest.json`
- `prepare_prompt.txt`
- `prepare_request.json`

适用场景：

- 检查 prompt 是否合理。
- 检查选中的章节和构件批次是否正确。
- 调试 Step2 输入而不真正调用模型。

#### 9.2.2 `step2-execute`

正式执行 Step2。

```bash
python3 -m pipeline_v2 step2-execute \
  --components data/input/components.json \
  --step1-source data/output/step1/run-20260407-step1 \
  --output data/output/step2/run-20260407-step2 \
  --component-batch-size 5 \
  --request-timeout-seconds 180 \
  --connection-retries 3
```

#### 9.2.3 `step2-synthesize`

当已有分章节或分批结果存在，但主结果丢失、异常或未正确收尾时，可以重新合成 Step2 主结果：

```bash
python3 -m pipeline_v2 step2-synthesize \
  --output data/output/step2/run-20260407-step2
```

### 9.3 Step2 常用参数

- `--components`
  - 必填，构件库路径，支持 `components.json` 或 `components.jsonl`。
- `--step1-source`
  - 必填，Step1 输出目录或 `chapter_index.json`。
- `--output`
  - 必填，Step2 输出目录。
- `--component-batch-size`
  - 每个章节窗口一次处理的构件批大小，默认 5。
- `--prepare-only`
  - 只写 prompt 和 manifest，不真正请求模型。
- `--no-resume-existing`
  - 不复用已完成批次，强制重跑。
- `--start-chapter-index` / `--chapter-limit`
  - 控制从哪个章节开始、处理多少章节。
- `--start-component-batch-index` / `--component-batch-limit`
  - 控制从哪个构件批开始、处理多少批。
- `--tpm-budget`
  - 本地节流预算。
- `--max-output-tokens`
  - 模型最大输出 token。
- `--validation-fallback-model`
  - 校验回退模型。
- `--validation-min-deviation-score`
  - 触发回退验证的偏差阈值。

## 10. Step3 使用说明

### 10.1 功能说明

Step3 是正式的“清单项 -> 构件类型 / 项目特征表达式 / 计算项目”匹配链路。它会综合：

- Step1 的 `table_regions.json`
- Step2 的 `component_matching_result.json`
- Step2 的 `synonym_library.json`
- 构件库 `components.json`

最终输出结构化中间表，作为后续人工复核和 Step4 的上游输入。

### 10.2 正式命令

```bash
python3 -m pipeline_v2 step3-execute \
  --step1-table-regions data/output/step1/run-20260407-step1/table_regions.json \
  --step2-result data/output/step2/run-20260407-step2/component_matching_result.json \
  --components data/input/components.json \
  --output data/output/step3/run-20260407-step3
```

### 10.3 常用参数

- `--config`
  - Step3 运行配置文件，默认读取 `pipeline_v2/step3_engine/runtime_config.ini`。
- `--step1-table-regions`
  - Step1 输出的 `table_regions.json`。
- `--step2-result`
  - Step2 主结果路径；也会用于自动推断同级 `synonym_library.json`。
- `--components`
  - 构件库路径。
- `--synonym-library`
  - 如自动推断不满足，也可显式指定。
- `--output`
  - 输出目录。
- `--prepare-only`
  - 只预处理和写 prompt，不发模型请求。
- `--local-only`
  - 只使用本地规则，不调用模型。
- `--max-rows-per-batch`
  - 每次 prompt 处理的最大表格行数。
- `--max-components-per-item`
  - 每个清单项保留的候选构件上限。

### 10.4 Step3 主要输出

Step3 输出目录通常包括：

- `project_component_feature_calc_matching_result.json`
- `run_summary.json`
- prompt 和批次中间文件
- 相关 Markdown 或 HTML 视图，视当前执行链而定

### 10.5 Step3 输出用途

- 用于人工复核。
- 用于生成 review queue。
- 用于 Step4 二次指定构件类型的直匹配。

### 10.6 生成人工复核队列

```bash

  --step3-result data/output/step3/run-20260407-step3/project_component_feature_calc_matching_result.json \
```

该命令会筛出：

- `candidate_only`

- `source_stage`
- `row_id`
- `quantity_component`
- `calculation_item_code`
- `review_status`
- `manual_notes`

## 11. Step4 使用说明

### 11.1 功能说明

Step4 用于在“构件类型已经确定”的前提下，继续为清单项补齐：

- 项目特征表达式
- 计算项目
- 计量单位
- 匹配依据与复核说明

它不会推翻指定构件类型本身，只会在该构件的属性、别名和计算项目范围内做本地直匹配和模型精修。

### 11.2 典型命令

#### 只跑本地直匹配

```bash
python3 -m pipeline_v2 step4-direct-match \
  --items-file data/input/sample_step4_items.json \
  --component-type 砼墙 \
  --components data/input/components.json \
  --synonym-library data/output/step2/run-20260407-step2/synonym_library.json \
  --local-only
```

#### 只准备提示词，不发模型请求

```bash
python3 -m pipeline_v2 step4-direct-match \
  --items-file data/input/sample_step4_items.json \
  --component-type 砼墙 \
  --config pipeline_v2/step4_runtime_config.ini \
  --prepare-only
```

#### 模型精修模式

```bash
python3 -m pipeline_v2 step4-direct-match \
  --items-file data/input/sample_step4_items.json \
  --component-type 砼墙 \
  --config pipeline_v2/step4_runtime_config.ini
```

#### 直接消费 Step3 结果

```bash
python3 -m pipeline_v2 step4-direct-match \
  --step3-result data/output/step3/run-20260407-step3/project_component_feature_calc_matching_result.json \
  --components data/input/components.json \
  --synonym-library data/output/step2/run-20260407-step2/synonym_library.json \
  --output data/output/step4/run-20260407-step4 \
  --local-only
```

### 11.3 Step4 常用参数

- `--items-file` / `--item-json`
  - 输入清单项 JSON。
- `--step3-result`
  - 直接消费 Step3 结果，自动按构件类型分组。
- `--component-type`
  - 指定构件类型。
- `--components`
  - 构件库路径。
- `--synonym-library`
  - 同义词库路径。
- `--config`
  - Step4 运行配置文件。
- `--output`
  - 输出目录。
- `--local-only`
  - 只跑本地直匹配。
- `--prepare-only`
  - 只写 prompt，不真正请求模型。
- `--max-items-per-batch`
  - 每批最多处理的清单项数量。

### 11.4 Step4 输出文件

典型输出包括：

- `step4_local_direct_match_result.json`
- `batch_001_prompt_input.json`
- `batch_001_prompt.txt`

- `batch_001_result.json`
- `step4_direct_match_result.json`
- `step4_direct_match_result.md`

### 11.5 Step4 结果字段

典型字段包括：

## 16. 常见问题与建议

### 16.1 Step2 中断后如何恢复

优先看：

- `run_summary.json`
- `chapter_run_index.json`
- 章节目录下的 `batch_*_result.json`

如果分批结果已经存在但主结果不完整，可以执行：

```bash
python3 -m pipeline_v2 step2-synthesize --output <step2_output_dir>
```

### 16.2 Step3 为什么找不到同义词库

Step3 会尝试根据 `--step2-result` 自动推断同级 `synonym_library.json`，但如果目录不标准或文件缺失，应显式补：

```bash
--synonym-library data/output/step2/<run>/synonym_library.json
```

### 16.3 Step4 什么时候用 `local-only`

推荐先用 `--local-only`：

- 无需模型密钥
- 可以先验证字段映射逻辑
- 更适合调试输入质量

待本地结果稳定后，再打开模型精修。

### 16.4 哪些目录不是正式入口

当前不是正式主线入口的目录包括：

- `分析工具/`
  - 主要作为本地环境和历史辅助目录保留。
- 历史报告中引用的旧目录
  - 只用于回溯，不作为现行执行入口。
