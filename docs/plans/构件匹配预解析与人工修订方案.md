# 构件匹配预解析与人工修订方案

## 1. 目标

当前已具备两类基础数据：

1. 构件列表数据
2. 国标 PDF 章节解析数据

下一步建议增加一个“预解析匹配层”，先用 AI 生成：

- 构件列表名称
- 国标文件中的实际构件名称
- 各类别名、俗称、简写、上下位叫法
- 匹配依据与置信度
- 待人工复核标记

该层定位为：

- 属于“分析流程中的预解析结果”
- 输出标准 JSON
- 后续可再接人工修订工具

人工修订工具定位为：

- 只作为结果补充修订入口
- 不纳入主分析流程
- 负责将 JSON 可视化、在线修改、再导回 JSON

## 2. 建议输入

建议将 AI 的输入分成 4 组：

### 2.1 构件列表

建议至少包含：

```json
[
  {
    "component_type": "砼墙",
    "source": "components.json",
    "properties": {}
  }
]
```

核心字段：

- `component_type`: 软件/构件列表中的构件名
- `properties`: 可选，用于辅助识别构件属性
- `source`: 可选，记录来源

### 2.2 国标章节解析结果

建议优先传入 `flat_regions.json` 或按章节整理后的精简结果，至少包含：

```json
[
  {
    "title": "附录G 混凝土及钢筋混凝土工程",
    "path_text": "附录G 混凝土及钢筋混凝土工程 > G.2 现浇混凝土墙",
    "text": "..."
  }
]
```

核心字段：

- `title`
- `path_text`
- `text`
- `tables`

### 2.3 可选人工先验词库

如果已有历史经验词库，建议同时输入：

```json
[
  {
    "canonical_name": "钢筋混凝土墙",
    "aliases": ["砼墙", "混凝土墙", "剪力墙"],
    "notes": "剪力墙需注意并非所有场景都等价"
  }
]
```

### 2.4 可选历史修订记录

如果已有人工确认结果，建议作为高优先级参考输入：

```json
[
  {
    "source_component_name": "砼墙",
    "selected_standard_name": "钢筋混凝土墙",
    "review_status": "confirmed"
  }
]
```

## 3. 建议输出 JSON 结构

建议 AI 输出统一结构，后续人工修订工具也以此结构为主：

```json
{
  "meta": {
    "task_name": "component_standard_name_matching",
    "standard_document": "房屋建筑与装饰工程工程量计算标准",
    "generated_at": "2026-03-19T00:00:00+08:00",
    "review_stage": "pre_parse"
  },
  "mappings": [
    {
      "source_component_name": "砼墙",
      "source_aliases": ["砼墙", "混凝土墙"],
      "selected_standard_name": "钢筋混凝土墙",
      "standard_aliases": ["钢筋混凝土墙", "混凝土墙", "现浇混凝土墙"],
      "candidate_standard_names": ["钢筋混凝土墙", "现浇混凝土墙"],
      "match_type": "alias_bridge",
      "match_status": "matched",
      "confidence": 0.93,
      "review_status": "pending",
      "evidence_paths": [
        "附录G 混凝土及钢筋混凝土工程 > G.2 现浇混凝土墙"
      ],
      "evidence_texts": [
        "章节中出现“现浇混凝土墙”“钢筋混凝土墙”等表述，与砼墙为高相关别名。"
      ],
      "reasoning": "构件列表中的“砼墙”为“混凝土墙”的行业简写，结合章节标题和正文表述，优先归并到“钢筋混凝土墙”。",
      "manual_notes": ""
    }
  ]
}
```

## 4. 匹配规则建议

建议 AI 严格按以下优先级进行判断：

### 4.1 直接同名匹配

- 构件列表名称与国标实际构件名称完全一致
- 置信度最高

### 4.2 标准名-别名桥接匹配

- 软件构件名是简称、俗称、口语化写法
- 国标章节中出现的是标准正式名称
- 例如：
  - `砼墙 -> 钢筋混凝土墙`
  - `现浇板 -> 混凝土板`
  - `构造柱 -> 混凝土构造柱`

### 4.3 上下位概念匹配

- 构件列表是更细颗粒度子类
- 国标中是更泛化的总称
- 例如：
  - `暗柱` 可能归入 `钢筋混凝土墙` 体系说明
  - `基础连梁` 可能归入 `基础梁` 或基础相关章节

### 4.4 章节上下文匹配

- 不能只看名字
- 还要结合所在章节、表格标题、项目特征、计量单位、计算规则
- 如果文本证据只支持“相关”但不支持“等价”，不能强行输出高置信度匹配

### 4.5 谨慎处理的情况

以下情况一律降低置信度，并标记人工复核：

- 一个软件构件可对应多个国标标准名
- 一个国标标准名可覆盖多个软件构件
- 仅通过单个字词相似判断
- 行业俗称有歧义
- 不同专业场景下同名异义

## 5. 推荐的状态字段

建议统一使用以下状态值：

- `match_status`
  - `matched`: 已有较明确匹配
  - `candidate_only`: 只有候选项，未最终确认
  - `unmatched`: 未找到合适国标名
  - `conflict`: 存在明显冲突

- `review_status`
  - `pending`: 待人工复核
  - `confirmed`: 人工已确认
  - `rejected`: 人工否决
  - `adjusted`: 人工已调整

- `match_type`
  - `exact`
  - `alias_bridge`
  - `contextual_inference`
  - `hierarchical`
  - `manual_override`

## 6. AI 提示词模板

下面这份提示词可直接用于“构件列表名称 -> 国标实际构件名称”预解析任务。

```text
你是一名建筑工程造价与国标清单解析专家，同时熟悉算量软件中的构件命名方式、行业简称、俗称和国标正式名称。

你的任务不是抽取工程量，也不是做最终业务裁定，而是做“构件名称预解析匹配”：
把输入的“构件列表名称”与“国标文件章节文本中出现的实际构件名称”建立尽可能可靠的匹配关系，并补充别名、候选项、证据、置信度与待复核标记。

你必须遵守以下原则：

1. 优先识别“正式名称-别名-简称-俗称”的对应关系。
2. 不能仅凭字面相似就强行匹配，必须结合章节上下文、标题、表格标题、项目特征描述、规则文本共同判断。
3. 如果只能判断“相关”，不能判断“等价”，则输出候选关系而不是强匹配。
4. 对一对多、多对一、上下位关系、专业歧义场景要显式标记。
5. 输出必须是合法 JSON，不要输出解释性散文，不要输出 Markdown。
6. 所有数组字段必须输出数组，不能把数组写成字符串。
7. `confidence` 取值范围为 0 到 1。
8. 若无把握，`review_status` 必须设为 `pending`。

输入数据如下：

【构件列表】
${COMPONENT_LIST_JSON}

【国标章节解析结果】
${STANDARD_REGION_JSON}

【可选先验别名字典】
${ALIAS_DICT_JSON}

【可选历史人工修订结果】
${HISTORY_REVIEW_JSON}

请执行以下步骤：

第一步：构建名称归一化视图
- 识别构件列表中的标准词、简称、俗称、口语化写法
- 识别国标章节中的正式构件名、子类构件名、体系名称
- 将“砼=混凝土”等行业常见简写纳入归一化判断，但不要过度泛化

第二步：为每个构件列表名称寻找候选国标名称
- 优先直接同名
- 其次别名桥接
- 再结合章节路径、正文、表格标题、项目特征和规则文本做上下文推断
- 对可能的多个候选项全部列出，并选择一个最优项；如果不能选择唯一最优项，则 `match_status` 设为 `candidate_only` 或 `conflict`

第三步：输出匹配结论
- 每条记录都要给出：
  - `source_component_name`
  - `source_aliases`
  - `selected_standard_name`
  - `standard_aliases`
  - `candidate_standard_names`
  - `match_type`
  - `match_status`
  - `confidence`
  - `review_status`
  - `evidence_paths`
  - `evidence_texts`
  - `reasoning`
  - `manual_notes`

第四步：严格按以下 JSON 结构输出

{
  "meta": {
    "task_name": "component_standard_name_matching",
    "standard_document": "<从输入内容概括文档名>",
    "generated_at": "<ISO8601时间>",
    "review_stage": "pre_parse"
  },
  "mappings": [
    {
      "source_component_name": "",
      "source_aliases": [],
      "selected_standard_name": "",
      "standard_aliases": [],
      "candidate_standard_names": [],
      "match_type": "exact|alias_bridge|contextual_inference|hierarchical|manual_override",
      "match_status": "matched|candidate_only|unmatched|conflict",
      "confidence": 0,
      "review_status": "pending|confirmed|rejected|adjusted",
      "evidence_paths": [],
      "evidence_texts": [],
      "reasoning": "",
      "manual_notes": ""
    }
  ]
}

补充要求：
- 如果某个构件找不到可靠对应项，仍然输出该构件记录，只是 `selected_standard_name` 置空，并将 `match_status` 设为 `unmatched`
- 不要漏掉任何输入构件
- 若章节文本里没有直接出现该构件名，但可通过上下位体系判断关联，请在 `reasoning` 中说明是“体系归并”还是“上下文推断”
```

## 7. 建议的调用方式

建议分两轮做：

### 7.1 第一轮：批量粗匹配

输入：

- 全量构件列表
- 国标章节精简结果
- 可选历史别名字典

输出：

- 全量预匹配 JSON

目标：

- 快速建立大部分稳定映射

### 7.2 第二轮：疑难项精修

输入：

- 第一轮中 `candidate_only`、`unmatched`、`conflict` 的记录
- 更完整的章节正文和表格内容

输出：

- 更高质量的二次修订 JSON

目标：

- 降低人工成本

## 8. 人工修订建议

建议人工主要关注：

- `confidence < 0.85`
- `match_status != matched`
- `review_status = pending`
- `candidate_standard_names` 数量大于 1
- `selected_standard_name` 为空

## 9. 与小工具的衔接建议

建议小工具直接读取上面这份 JSON：

- `meta` 保留原样
- `mappings` 进入表格编辑
- 数组字段在界面中按“每行一个值”或“分号分隔”展示
- 人工调整后重新导出为同结构 JSON

这样可以保证：

1. AI 预解析可独立运行
2. 人工修订可后置补充
3. 主流程与人工工具解耦
4. 后续如要沉淀词库，也可以直接从修订后的 JSON 回写
