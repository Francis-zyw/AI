1. 项目背景与建设目标

国标解析-文本分析的核心目标，不是只完成一次性的文档抽取，而是把国标 PDF 中分散的章节结构、表格信息、构件名称、项目特征和计算项目，转化为一条可持续优化、可追踪、可复用的知识生产链路。随着功能逐步完善，我们的重点已经从“把模型接进来”转向“如何让整条处理链路更稳定、更高效、更能沉淀经验”。

基于现有功能，多Agent协同方案的价值已经比较明确：通过主脑小凯统一调度多个子 Agent，把原本依赖长对话和人工记忆的复杂工作，拆成分阶段、可留痕、可回写的工程化流程。这样不仅能提升当前任务的处理效率，也能让后续同类国标文档越跑越稳、越用越熟。

这份总结也带有明显的复盘性质。相比之前单 Agent 长对话模式下容易出现的上下文漂移、重复犯错、弱匹配被提前当成正式结果、人工纠正无法长期沉淀等问题，多Agent协同方式更强调角色分工、过程留痕、质量审核和记忆回写。

2. 多Agent协同方案总体架构

多Agent协同方案是一套面向国标解析的知识生产流水线。它由主脑小凯负责任务拆解、状态编排、结果验收与记忆回写，子 Agent 分阶段完成文档结构解析、构件归一、候选匹配和结果精修，并通过工作区隔离、review queue 和人工复核闭环持续优化。

主脑小凯承担总调度与总验收职责，是多 Agent 协同体系中的控制平面。

从工程视角看，多Agent协同方案可以拆成五层：

  1. 输入层：PDF 国标文件、构件库、历史人工修订结果。
  2. 主脑调度层：负责审计现状、拆解任务、分配子 Agent、组织阶段输入输出、回收结果并触发二次验证。
  3. 子 Agent 执行层：负责 Step1、Step2、Step3、Step4 等分阶段能力。
  4. 记忆层：负责工作区留痕、同义词库、review queue、manual review ledger 和长期知识沉淀。
  5. 发布层：负责形成可复核、可追溯的正式结果与后续优化依据。

3. 技术路线与分阶段流程

3.1 Step1：正文范围识别、章节切分与表格区域提取

Step1 的定位不是直接做完整清单字段解析，而是先把原始国标 PDF 预处理成可供后续步骤消费的章节级结构化数据底座。现有技术路线分为四段：

  1. 根据目录或页签定位正文有效范围，剔除封面、版权页、公告、前言、目录等前置无效内容。
  2. 基于目录树递归切分正文，生成区域树、扁平区域和顶级章节分包。
  3. 在每个区域内区分说明性文本和表格块，单独抽取表格相关字段。
  4. 输出 catalog_summary.json、flat_regions.json、chapter_regions/chapter_index.json、table_regions.json 等标准产物，供后续步骤直接消费。

识别策略上采用“文本层优先，OCR 兜底”。对于当前样本文档，主路径走文本层提取，OCR 作为后续扫描件、图片页和弱文本层页面的扩展入口保留。

3.2 Step2：构件归一、同义词库沉淀与章节证据组织

Step2 负责把软件构件名、国标标准名称、行业别名和章节证据组织起来，生成结构化的构件匹配结果和同义词库。这一步的目标不是简单输出一个名称，而是沉淀出可复核、可回写、可长期复用的名称归一体系。

Step2 的核心产物包括 component_matching_result.json 和 synonym_library.json。其中同义词库不仅用于当前匹配，还用于后续规则匹配、人工复核与长期知识沉淀。系统会同时保留 review_status、evidence_paths、evidence_texts、reasoning 等字段，保证结果既能看结论，也能看依据。

3.3 Step3：清单项到候选构件、项目特征和计算项目的匹配

Step3 负责把 Step1 的清单主数据、Step2 的同义词库和构件属性库整合起来，生成“清单项 - 候选构件 - 项目特征表达式 - 计算项目”的中间结果。这一步不再追求一次性给出所有正式答案，而是把不确定性显式保留下来。

目前 Step3 会把结果区分为 matched、candidate_only、unmatched 等状态。这样做的核心思想是：允许不确定，拒绝伪确定。低置信度结果先进入待复核链路，而不是过早包装成正确答案。

3.4 Step4：指定构件类型后的直匹配与模型精修

Step4 面向“上游已经明确指定构件类型”的场景，在固定构件范围内继续生成项目特征表达式、计算项目和计量单位。它保留本地直匹配能力作为基线，同时支持可选模型精修，用于进一步校正特征表达式、计算项目和结果说明。

从实施价值上看，Step4 让整个链路具备了“先粗匹配，再局部精修”的能力，既能控制模型使用范围，也能提升特定构件场景下的结果质量。

3.5 审计、review queue 与验证门

多Agent协同方案并不是只关心产出，还关心产出的可信度和可交付性。因此在 Step1 到 Step4 之外，我们补充了审计、review queue 和验证门控能力。主脑小凯会先审计当前状态，再组织阶段任务、回收中间结果，对低置信度、冲突项和未匹配项统一进入 review queue，最后结合人工抽检和 cutover gate 做二次验证。

4. 核心机制设计

4.1 主脑调度

小凯不是单纯的概念化助手，而是整条处理链路的状态控制中心。它不直接替代所有业务推理，而是负责审计现状、拆解任务、分配子 Agent、组织阶段输入输出、回收中间结果、触发 review queue、执行二次验证，并将人工确认结果回写为长期记忆，驱动后续任务持续优化。

4.2 分运行工作区隔离

每次运行在独立工作区内完成，工作区中会保留本轮输入、提示词、模型输出、结果、错误信息和运行摘要。这样做的价值在于：新任务不会被旧错误污染，失败批次可以局部续跑，整个处理过程可以回放和审计，经验也能沿着工作区和结果产物沉淀下来。

4.3 Review Queue

多Agent协同方案不把所有结果一股脑当成“完成”，而是把 candidate_only、unmatched、conflict 等不确定结果组织成 review queue。这样人工复核不再是全量返工，而是聚焦到最有价值、最需要判断的部分。

4.4 Manual Review Ledger 与长期记忆回写

人工复核结果不会只停留在聊天记录里，而是要写入标准化的记录中，包括 record_id、review_status、manual_notes、reviewed_at 等字段。后续这些结果会反哺构件匹配结果、同义词库和最终导出结果，形成长期知识记忆。

4.5 二次验证与 Cutover Gate

子 Agent 完成任务后，主脑不会直接相信第一次输出，而是通过审计报告、基线对比、人工抽检和 cutover gate 做二次验证。这样可以把“子 Agent 说做完了”和“系统真的可以交付”这两件事区分开来，保证整条链路的稳定性。

4.6 多 Agent 角色分工说明

在多Agent协同方案中，多 Agent 不是简单地把任务拆给多个模型去做，而是形成了一套有明确分工的协同机制。可以把它理解为“协调者主脑 + 执行者子 Agent + 审核者验证 + 记忆回写”的闭环。

角色 职责定位 在多Agent协同方案中的作用
主脑小凯 / 协调者 负责理解目标、拆解任务、组织流程、分配子 Agent、回收结果并决定下一步动作 小凯承担总调度与总验收职责，是整条处理链路的控制中心，不直接替代所有业务推理，而是负责让每一步按正确顺序、正确状态推进。
执行者子 Agent 负责具体业务子任务执行 例如 Step1 做正文范围识别与章节切分，Step2 做构件归一与同义词沉淀，Step3 做候选匹配，Step4 做指定构件后的直匹配与模型精修。每个子 Agent 只处理自己边界内的问题，减少长上下文干扰。
审核者 / 验证者 负责检查结果质量，不直接生产业务结论 审核者重点检查是否存在幻觉、遗漏、逻辑冲突、格式问题和低置信度结果，并结合 review queue、人工抽检和 cutover gate 决定哪些结果可以进入正式产物，哪些需要打回重做或进入人工复核。
记忆回写者 负责把人工修订和复核结论沉淀为长期知识 通过 manual review ledger、同义词库、结果回写等方式，把“这次修正过的经验”变成“下次可直接复用的系统记忆”，避免重复犯同样的错误。

从最简闭环看，这套机制可以概括为：主脑协调，子 Agent 执行，审核者验证；从完整工程闭环看，则是：主脑协调，子 Agent 执行，审核者验证，人工复核回写，长期记忆沉淀，最终形成正式发布结果。

也正因为这样，多Agent协同方案的优势不只是“有多个 Agent”，而是“不同角色各司其职”。系统不再是一个 Agent 既干活又自己给自己打分，而是由主脑组织分工，由执行者完成任务，由审核者把关质量，由记忆回写机制负责让系统持续进化。

6. 当前成果与优化心得

6.1 当前成果

从现有产物看，多Agent协同方案已经具备清晰的阶段化能力和工程化雏形：

  • Step1 已将样本文档处理为结构化底座：总页数 122 页，正文区域 119 个，含表格区域 105 个，表格行数 1290 条。
  • Step2 已形成“构件匹配结果 + 同义词库”的产物形态，为后续长期记忆提供基础。
  • Step3 已形成清单项到候选构件、项目特征和计算项目的中间结果结构，并显式保留不确定状态。
  • Step4 已支持指定构件类型后的直匹配与模型精修，补强局部高精度场景。
  • V2 已形成工作区留痕、运行摘要、review queue、人工复核回写和验证门控的整体思路。

6.2 优化心得

这套方案带来的最大变化，不是“模型更会回答”，而是“系统更会协作”。从实施过程回看，我们有几条非常明确的心得：

  1. 先做边界，再做内容。 先确定正文范围和章节边界，才能减少噪声输入，稳定后续模型调用。
  2. 先做切分，再做理解。 把整本文档拆成章节包和任务块后，模型的稳定性、可控性和成本控制都更好。
  3. 先做结构，再做智能。 表格、说明文字、章节层级先理顺，后续构件匹配和规则提取才有可靠输入。
  4. 允许不确定，拒绝伪确定。 对低置信度结果先挂起、再复核、再回写，比硬猜一个答案更稳。
  5. 纠错必须写进系统，不只写在对话里。 只有把人工修正沉淀成长期记忆，系统才会真正持续进化。
多Agent协同方案的核心，不是把模型接入流程，而是把流程重构成可分工、可回放、可持续进化的知识生产线。

7. 后续优化方向

  • 进一步跑通 Step2 到 Step4 的正式闭环，持续降低 candidate_only 和 unmatched 的比例。
  • 扩大长期记忆的复用范围，让历史人工修订结果在后续相似文档中直接生效。
  • 继续增强对扫描件、图片页、无有效目录文档的处理能力。
  • 进一步完善 review queue 与 manual review ledger 的回写链路，让人工复核结果更快反哺主线。
  • 持续优化外部模型配置、运行文档和交付方式，保证方案可复用、可迁移、可上传。

8. 总结

从现有功能看，多Agent协同方案已经不再是单一模型问答式的应用，而是一套面向国标解析的多 Agent 协同知识生产体系。主脑小凯负责调度，子 Agent 负责执行，人工复核负责校正，长期记忆负责沉淀，这套机制共同保证了方案能够持续优化、持续收敛、持续复用。

因此，这套方案的价值不只体现在单次命中率，更体现在人效、算效和长期复用能力的同步提升。随着工作区隔离、review queue、人工复核回写和验证门的持续完善，多Agent协同方案将进一步从“会回答”升级为“会协作、会记忆、会复盘、会自我校验”的高能效智能协同方案。
