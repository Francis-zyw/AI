## 1. 这次重做的目标

本次不再继续在旧版 `step1_chapter_ocr/`、`step2_component_match/`、`step3_filter_condition_match/` 上叠加修补，而是：

1. 先完整分析现有项目功能与失败点

## 2. 当前项目功能版图

当前项目主线实际包含 3 个阶段和 2 个辅助工具：

- `step1_chapter_ocr/`
  - 负责 PDF 目录识别、正文范围定位、章节切分、表格区域与表格行抽取
  - 当前真实输出已经完整落在 `data/output/step1/房屋建筑与装饰工程工程量计算标准/`

- `step2_component_match/`
  - 负责把 `components.json` 与 Step 1 章节包进行构件预匹配，目标是生成 `component_matching_result.json` 和 `synonym_library.json`
  - 当前真实运行没有完成正式结果

- `step3_filter_condition_match/`
  - 负责把 Step 1 清单行、Step 2 同义词库、构件属性库整合成 “清单-构件-项目特征表达式-计算项目” 中间表
  - 当前真实结果只有本地规则版，没有完成模型校正闭环

- `tools/tool_component_type_library/`
  - 负责维护构件类型-属性库

- `tools/tool_component_match_review/`
  - 负责人工复核 Step 2 结果

## 4. 多 Agent 分工方案

本轮已按以下职责分工：

1. `总agent`
   - 汇总项目结构、运行产物、真实失败点
   - 设计 V2 主流程、工作单元和隔离策略
   - 新增 `pipeline_v2/` 审计与重构骨架

2. `Step1 agent`
   - 只读分析 Step 1 的模块、输入输出、测试与结构缺陷
   - 结论：Step 1 应保留输出契约，但必须拆掉 TOC 强耦合和 OCR 伪兜底

3. `Step2 agent`
   - 目标职责：分析批处理、窗口切分、模型调用与失败恢复
   - 当前总agent已确认的关键结论：失败点是请求格式，不是批量窗口策略本身

4. `Step3 agent`
   - 目标职责：分析本地规则、模型校正、人工复核链路
   - 当前总agent已确认的关键结论：Step 3 只有 local_only 结果，复核积压很大

5. `数据/工具 agent`
   - 目标职责：识别重复目录、旧代码副本、数据与脚本错位
   - 当前总agent已确认的关键结论：构件库副本过多，主线入口需要重新收口

## 5. 新的整体处理流程

V2 重新定义后的主流程如下：

```text
PDF / 构件库 / 历史修订结果
    ↓
V2 Workspace Init
    ↓
Step 1: Layout & Chapter Extraction
    ↓
Step 2: Component Alias Alignment
    ↓
Step 3: Candidate Matrix Generation
    ↓
Model Refine
    ↓
Review Queue
    ↓
Confirmed Export
```

### 核心变化

- 每次运行先建立 `data/workspaces_v2/<run_id>/`
- 每一步都保留：
  - `inputs.json`
  - `manifest.json`
  - `prompt.txt`
  - `model_output.txt`
  - `result.json`
  - `error.txt`
- `candidate_only / unmatched / conflict` 不再混在最终结果里装作已完成，而是进入专门的 review queue
- 旧版 `data/output/step1|2|3/` 保持只读参考，不再作为新主线写入目标

### WU-01 冻结旧主线并建立 V2 隔离区

  - 不再把新改动继续写进旧主线
  - 明确后续开发从 `pipeline_v2/` 开始

### WU-02 重写 Step 1 契约层

### WU-03 重写 Step 2 请求与结果编排

- 目标：
  - 替换旧版请求格式
  - 让批处理与失败恢复可续跑

### WU-04 重写 Step 3 为三段式状态机

- 目标：
  - `local_rule -> model_refine -> review_queue -> confirmed_export`

### WU-05 建立端到端审计与验收

- 这个编号在初版方案里偏早，现已后移为最终验收单元

### WU-05 迁移 legacy 产物与人工修订

- 目标：
  - 把已有的 Step 1 可用输出、Step 2 人工修订 JSON、Step 3 候选结果导入 V2
- 要求：
  - 每份导入都要记录来源路径、时间、校验信息
  - 导入后的产物写入 `data/workspaces_v2/<run_id>/imports/`
  - 人工修订结果写入 `data/manual_reviews/step2/` 或 `data/manual_reviews/step3/`

### WU-06 定义人工复核契约与回写链路

- 目标：
  - 让人工确认结果真正反哺 Step 2 / Step 3，而不是停留在独立工具里
- 最低字段要求：
  - `record_id`
  - `source_stage`
  - `review_status`
  - `manual_notes`
  - `reviewed_at`
- 回写方向：
  - Step 2 回写 `synonym_library.json / component_matching_result.json`
  - Step 3 回写 `review queue / final export`

### WU-07 建立验证门与 cutover 标准

- 目标：
  - 明确 V2 何时只能 shadow run，何时可以替换旧主线
- 核心标准：
  - Step 2 必须加载全量附录章节，而不是默认首章
  - 正式模式下禁止出现 `completed_local_only`
  - baseline 文档上 `matched_rows` 不低于当前 v1 基线 `198`
  - `candidate_only + unmatched` 总量相对当前基线至少下降 `20%`，或全部进入可追踪 review queue
  - 人工抽检不少于 `50` 行或总量 `10%`，取较大值，准确率达到 `90%+`

### WU-08 建立端到端审计与验收

- 目标：
  - 每次运行都能自动告诉我们：
    - 哪一步成功
    - 哪一步失败
    - 哪些数据待复核
    - 下一个 agent 应该接什么任务

## 10. Cutover Gate

### Gate 1: Shadow Run

- baseline 文档必须成功产出：
  - `chapter_index.json`
  - `table_regions.json`
  - `synonym_library.json`
  - `review queue`
- 不允许出现：
  - Step 2 request schema error
  - Step 3 正式模式仍是 `completed_local_only`

### Gate 2: Default Cutover

- baseline 文档的 `matched_rows` 不低于当前 v1 基线 `198`
- `candidate_only + unmatched` 较当前基线下降至少 `20%`，或全部进入 review queue
- 人工抽检：
  - 至少 `50` 行或总量 `10%`
  - 准确率达到 `90%+`
- 只有通过上述门禁，V2 才能替换旧主线
