# 国标清单文件智能处理 - 完整实施方案

## 一、核心问题解答

### Q: 是否要先进行文本挖掘并转换为JSONL格式？

**A: 建议采用"渐进式处理流程"：**

```
原始文档 → 中间结构化数据 → JSONL → 知识图谱/数据库
   ↓            ↓            ↓           ↓
 PDF/图片    DataFrame    标准交换格式  查询分析
             (便于清洗)   (便于存储)   (便于应用)
```

**理由：**
1. 中间使用pandas DataFrame便于数据清洗、去重、验证
2. JSONL作为标准交换格式，每行独立便于流式处理
3. 最终可导入Neo4j等图数据库支持复杂关联查询

## 二、完整处理流程

### 阶段1: 输入处理

| 输入类型 | 处理方式 | 工具 |
|---------|---------|------|
| PDF文件(有文本层) | 直接提取文本+坐标 | pdfplumber, PyMuPDF |
| 扫描图片(无文本层) | OCR图文识别 | PaddleOCR, EasyOCR |
| 其他格式(Word等) | 格式转换后处理 | python-docx等 |

### 阶段2: 版面分析与区域定位

```
┌─────────────────────────────────────────┐
│              版面分析器                  │
│           Layout Analysis               │
└──────────────────┬──────────────────────┘
                   │
    ┌──────────────┼──────────────┐
    ▼              ▼              ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│章节标题区│  │表格区域 │  │说明文本区│
│(A.1 xxx)│  │(清单明细)│  │(适用范围)│
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     ▼            ▼            ▼
提取章节编码   提取表格结构   文本挖掘分析
章节名称       清单编码/名称  关键词提取
               项目特征       构件关联
```

### 阶段3: 结构化数据提取

**数据模型：**

```
Chapter (章节) ──1:N──> Item (清单项)
   │                       │
   │                       N:1──> Feature (项目特征)
   │                       │          │
   N:M                     │          N:M──> Keyword
   ↓                       ↓          (关键词)
Component               Component
(构件类型)              (关联构件)
```

**提取字段：**
- **章节**: chapter_code, chapter_name, description
- **清单项**: item_code, item_name, unit, calc_rule
- **项目特征**: feature_name, feature_value, keywords
- **构件类型**: component_name, category, confidence

### 阶段4: 文本挖掘与关联分析

**技术栈：**

1. **中文分词**: jieba / pkuseg
   - 添加建筑领域自定义词典

2. **关键词提取**: TF-IDF / TextRank
   - 提取高频专业术语

3. **命名实体识别**: 规则 + 词典匹配
   - 构件类型: 建筑物、构筑物、桩基础...
   - 材料类型: 混凝土、钢筋、砖...
   - 工艺类型: 开挖、浇筑、铺设...

4. **关联规则挖掘**:
   - "适用于" → 章节-构件关联
   - "项目特征包括" → 特征-关键词关联

### 阶段5: 输出与存储

| 格式 | 用途 | 特点 |
|------|------|------|
| JSONL | 推荐 | 每行独立，便于流式处理，易于扩展 |
| CSV | 查看用 | Excel打开，便于筛选和统计 |
| Neo4j | 图数据库 | 关联查询，可视化，深度分析 |

**JSONL Schema:**
```json
{"type": "chapter/item/relation/component/feature", "data": {...}}
```

## 三、关键技术选型

| 功能模块 | 推荐方案 | 备选方案 |
|---------|---------|---------|
| PDF解析 | pdfplumber | PyMuPDF, Camelot |
| OCR识别 | PaddleOCR | EasyOCR, Tesseract |
| 表格识别 | PP-Structure | TableTransformer, Camelot |
| 版面分析 | 自定义规则+坐标 | LayoutParser, Detectron2 |
| 中文分词 | jieba | pkuseg, HanLP |
| 关键词提取 | TF-IDF/TextRank | LDA, BERT-based |
| 实体识别 | 规则+词典 | 训练NER模型, 大模型抽取 |
| 知识存储 | Neo4j | MongoDB, PostgreSQL+pgvector |

## 五、实施建议

### 1. 分阶段实施
- **第一阶段**：实现基础OCR和表格提取
- **第二阶段**：添加文本挖掘和关键词提取
- **第三阶段**：构建知识图谱和关联分析

### 2. 质量控制
- 设置置信度阈值（如OCR置信度<0.8标记待审核）
- 建立人工审核机制
- 持续优化词典和规则

### 3. 性能优化
- 使用多线程并行处理
- 对大规模文档使用批处理
- 考虑使用GPU加速OCR

### 4. 扩展性考虑
- 设计可配置的规则引擎
- 支持多种国标规范
- 预留大模型接口用于复杂抽取


```json
{"type": "node", "data": {...}}

```

## 七、使用示例

```bash

# 3. 查看输出结果
ls data/output/

# chapters.jsonl        # 章节信息
```
